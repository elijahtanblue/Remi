# Remi

Remi reconstructs the real status of an issue when the important evidence is scattered across Slack, Jira, and Gmail. The core artifact is the **Current Work Record**: an issue-scoped view of what is happening now, what changed, who owns the next step, what is blocked, and which evidence supports that read.

Remi is not broad workspace search, a generic summarizer, or a Slack helper. It is a coordination layer for messy issues where the ticket alone is not the whole story.

## Current Product Direction

[Remi-ticket-reconstruction-assistant-v3.md](../design/Remi-ticket-reconstruction-assistant-v3.md) is the singular product source of truth. This README is an implementation-facing summary of that direction.

Remi is moving toward a guided, private-beta product for support, implementation, customer operations, and escalation-heavy workflows. The wedge is issue-scoped reconstruction: show the real blocker, owner, and next action for work where Jira, Slack, and email each contain only part of the truth.

The long-term product home is the Remi web workflow:

- **Web product:** work queue, issue detail page, Current Work Record, evidence timeline, approval inbox, and scoped workflow settings
- **Slack:** action surface for linking context, requesting a brief, generating a draft doc, and receiving nudges
- **Jira:** anchor system for the issue identity, status, ownership, and controlled writeback proposals
- **Gmail:** core evidence source for customer, vendor, and escalation context

Remi should stay issue-scoped and workflow-scoped. It should not become company-wide search, an enterprise knowledge base, or an autonomous agent that silently mutates customer systems.

## Current Implementation vs Roadmap

The deployable v0.x implementation is still ahead in some places and behind in others.

**Implemented today:**

- Slack thread to Jira issue linking
- deterministic `/brief` output from Slack and Jira events
- Jira issue sidebar panel
- legacy Slack overview as a current, secondary surface
- Gmail mailbox monitoring for Google Workspace
- AI-backed memory snapshots and approval-gated Jira comment proposals
- write-only Confluence page generation from issue context
- admin/operator dashboard for installs, summaries, memory, analytics, and failed jobs

**Roadmap direction:**

- Current Work Record becomes the primary product object
- the web product becomes the home of day-to-day workflow review
- `/brief` and Jira panels read from the Current Work Record rather than treating legacy brief generation as the center of the product
- AI updates remain controlled, reviewable, cited, and scoped to an issue/workflow
- scope boundaries evolve from department-style labels toward a flexible `Scope` model that can represent a team, project, workflow, department, or pilot boundary

## Guided Pilot Motion

Remi is currently best framed as a guided design-partner pilot, not a broad self-serve install.

Pilot shape:

- one workflow or support/implementation queue
- one narrow scope of Slack channels, Jira projects, and Gmail mailboxes
- design-partner onboarding and private app installs
- clear success metrics: fewer context-hunting loops, faster blocker identification, cleaner handoffs, and more reviewable next actions

Operator setup still lives in [SETUP.md](SETUP.md), but those instructions describe the current deployable system rather than the market-facing product motion.

## Current User Flows

### Link an Issue

Use `/link-ticket ISSUE-KEY` inside the Slack thread where the real discussion is happening. Remi links that thread to the Jira issue, backfills available history, and stores future updates against the issue.

### Get the Current Brief

Use `/brief ISSUE-KEY` to ask Remi for the current read on an issue. In v0.x this may still be generated by the rules-based brief package or the newer Current Work Record path, depending on workspace configuration.

### Generate a Confluence Draft

Use `/doc ISSUE-KEY [handoff|summary|escalation]` to generate a write-only Confluence page from the issue context Remi already knows about. Confluence is a near-term docs target, but Remi should not be repositioned as a documentation product.

Current constraints:

- Confluence generation is write-only first
- Slack-triggered generation comes first
- no macro-heavy or Forge-native Confluence UI in V1
- Notion is deferred until Confluence is validated

### Review AI Proposals

AI-backed updates are controlled through proposals. Remi can draft Jira comment writebacks from the issue context, but users or operators must approve them before anything is applied externally.

---

## Contextual Uploads

Contextual document uploads are planned as **pinned issue context**: project briefs, reference docs, requirements, or customer materials that help Remi understand why an issue matters.

They are trusted inputs, not higher truth. Uploaded documents should enrich retrieval and generation, but they should not automatically override live Jira, Slack, or Gmail evidence. The retrieval path should select relevant uploaded content for the issue rather than stuffing every uploaded document into every prompt.

Near-term constraints:

- issue-scoped upload first
- reference-doc use case first
- text PDF, Markdown, and plain text first
- scanned/image PDFs remain out of scope
- global workspace document library remains out of scope

---

## Product Boundaries

Remi should keep these boundaries crisp:

- issue-scoped, not company-scoped
- Current Work Record, not one-off answer
- coordination and next action, not generic productivity
- support, implementation, customer ops, and escalation workflows before generic PM productivity
- controlled writeback, not silent automation
- guided pilot, not open-ended workspace crawl

## Monorepo Structure

```text
apps/
  api/        Fastify API server (Slack, Jira, admin routes)
  worker/     background consumers for Slack, Jira, Gmail, memory, docs
  admin/      current operator dashboard

packages/
  shared/         types, schemas, constants, errors
  db/             Prisma schema, client, repositories
  queue/          queue abstraction (SQS in prod, memory in dev)
  storage/        storage abstraction (S3 in prod, local in dev)
  slack/          Slack handlers, commands, views, OAuth
  jira/           Jira Connect auth, REST client, webhooks, panel
  gmail/          Gmail sync, issue detection, Slack DM notifications
  confluence/     Confluence REST client, IssueDocContext builder, page renderer
  memory-engine/  AI extraction, snapshot, and proposal pipeline
  summary-engine/ rules-based brief generation retained as fallback
```
